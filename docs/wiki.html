Getting Started
What is Turnstil?

Turnstil is a QR-based event check-in and contact-sharing platform. Each user gets a permanent QR code tied to their identity, used for both event attendance and networking. No per-event tickets, no disposable codes — one identity, everywhere.

✦
Universal QR Identity
One permanent QR code per person — works across all events and contact sharing.
◈
Live Check-In
Camera-based scanning with instant validation, walk-in support, and audit logs.
◐
Discovery Mode
Scan anyone's QR to see their contact card — privacy controls let users choose what's visible.

Start the stack
docker compose up -d
Run migrations
docker compose exec turnstil python manage.py migrate
Create an admin user
docker compose exec turnstil python manage.py create_admin --username admin --password admin1234 --email [email protected] --name "Admin User"

Docker Setup
Turnstil is distributed as a Docker image on Docker Hub. The stack includes the Django application server and a PostgreSQL database.

Prerequisites

  • Docker Engine 20.10+ and Docker Compose v2 (or Portainer Stacks)
  • At least 512 MB free RAM and 1 GB disk space

docker-compose.yml

This is the reference compose file. It pulls the published image from Docker Hub, connects to a PostgreSQL 16 database, exposes the app on port 9100, and runs a background reminder service for email notifications. Environment variables are shared via a YAML anchor to avoid duplication.

YAML 
# Shared environment variables (YAML anchor)
x-shared-env: &shared-env
  - DEBUG=True
  - SECRET_KEY=dev-secret-change-in-prod-make-this-long-and-random
  - ALLOWED_HOSTS=localhost,127.0.0.1,your-domain.com
  - CSRF_TRUSTED_ORIGINS=https://your-domain.com,http://localhost:9100
  - DATABASE_URL=postgresql://turnstil:turnstil@db:5432/turnstil
  - DB_HOST=db
  - DB_NAME=turnstil
  - DB_USER=turnstil
  - DB_PASSWORD=turnstil
  - EMAIL_HOST_PASSWORD=${EMAIL_HOST_PASSWORD:-}

services:
  turnstil:
    image: connorhaggerty/turnstil:latest
    container_name: turnstil
    restart: unless-stopped
    ports:
      - "9100:8000"
    environment: *shared-env
    volumes:
      - turnstil_media:/app/media
    depends_on:
      - db

  reminders:
    image: connorhaggerty/turnstil:latest
    container_name: turnstil-reminders
    restart: unless-stopped
    command: python manage.py send_reminders --loop
    environment: *shared-env
    depends_on:
      - db

  db:
    image: postgres:16-alpine
    container_name: turnstil-db
    restart: unless-stopped
    environment:
      - POSTGRES_USER=turnstil
      - POSTGRES_PASSWORD=turnstil
      - POSTGRES_DB=turnstil
    volumes:
      - turnstil_db:/var/lib/postgresql/data

volumes:
  turnstil_db:
  turnstil_media:
!
For production, generate a strong SECRET_KEY, set DEBUG=False, use a unique DB_PASSWORD, and configure ALLOWED_HOSTS and CSRF_TRUSTED_ORIGINS for your domain.

Portainer Deployment

If you're running Portainer, paste the compose file above into Stacks → Add stack and deploy. The container names (turnstil and turnstil-db) are set explicitly so they're easy to identify.

Management Commands
Turnstil includes custom Django management commands for user creation and demo data seeding. Run them via docker compose exec turnstil python manage.py <command>.

create_admin

Creates an admin user with full system access (admin role, is_staff and is_superuser flags set). Also creates the linked Person record.

python manage.py create_admin \
  --username admin \
  --password admin1234 \
  --email [email protected] \
  --name "Admin User"
ArgumentRequiredDescription
--usernameYesLogin username
--passwordYesAccount password
--emailYesEmail address
--nameYesDisplay name (on Person record)

create_user

Creates a regular user with the attendee role. Same arguments as create_admin.

python manage.py create_user \
  --username jdoe \
  --password changeme123 \
  --email [email protected] \
  --name "Jane Doe"

send_reminders

Sends reminder emails to registered attendees before their events start. Checks configurable reminder windows (default: 24 hours and 1 hour before event start). Uses the EventReminder model to track which reminders have already been sent, preventing duplicates.

# Run once
python manage.py send_reminders

# Run as a daemon (checks every 15 minutes)
python manage.py send_reminders --loop
FlagDescription
--loopRun continuously, checking every 15 minutes
✓
The Docker compose file includes a dedicated reminders service that runs send_reminders --loop automatically. No cron setup needed.

seed_demo

Populates the database with a full set of demo data for testing: an admin, an organizer, a staff user, five attendees, three events, and registrations. Safe to run multiple times — it uses get_or_create and skips existing records.

Email Notifications

Turnstil sends email notifications for event-related activities:

  • Event creation — the event owner receives a confirmation email when creating a new event
  • Event editing — the event owner receives notification when their event is updated
  • Registration confirmation — attendees receive an email when they register for an event
  • Reminder emails — registered attendees receive automatic reminders at 24 hours and 1 hour before an event starts

Email delivery requires setting EMAIL_HOST_PASSWORD to a valid SMTP credential (e.g. a Gmail App Password). Without it, emails are printed to the console — useful for development and testing. All send_mail calls are wrapped in try/except so email failures never crash the application.

✓
To test email locally without SMTP, leave EMAIL_HOST_PASSWORD unset. Django's console backend will print email content to stdout so you can verify the format.
Features
Authentication & Accounts

Registration

Users create an account by providing the following fields:

  • Username — must be unique
  • Email — must be unique
  • Password — minimum 8 characters
  • Full name — displayed on their contact card and QR display
  • Organization — optional, shown on contact card if visibility is enabled

Registration creates two linked records: a User (for login credentials and role) and a Person (for identity, contact info, and QR code). The Person is assigned a permanent UUID that becomes the payload of their QR code. New accounts default to the attendee role.

After registering, users are automatically logged in and redirected to their profile.

Login / Logout

Session-based login is used for all web pages. After login, users are redirected based on their role:

  • Staff, organizer, admin → redirected to the scanner page (/scanner/)
  • Attendee → redirected to their profile (/profile/), or to the ?next= URL if one was provided

Logging out clears the session (including any active event selection) and returns to the home page.

JWT Authentication (API)

API consumers authenticate via JSON Web Tokens (SimpleJWT). Login returns an access token and a refresh token. The scanner page stores the JWT in localStorage to make authenticated API calls from the browser.

TokenLifetimeNotes
Access token8 hoursInclude as Authorization: Bearer <token>
Refresh token7 daysTokens are rotated on refresh
i
The scanner page uses JWT for API calls rather than session auth, allowing it to make direct fetch() requests to the REST endpoints.

Home Page

The public home page (/) displays all upcoming events as cards. Each card shows the event name, date/time, location, and a link to the event detail page. Unauthenticated visitors can browse events and are prompted to log in or register to sign up.

Identity & QR Codes

Every user has a Person record with a permanent UUID. This UUID is the payload of their QR code — it never changes, is not tied to any specific event, and serves as the user's universal identifier across the platform.

QR Code Display

  • Profile page — shows a thumbnail of the QR code with a link to the fullscreen view.
  • Fullscreen view (/profile/qr/) — displays the QR code at full size along with the user's name and organization. Includes a download button that saves the QR as a PNG file.

QR Code Generation

QR codes are generated on demand as PNG images via the API at /api/people/<uuid>/qr. The QR encodes only the Person UUID — no personal data is embedded in the code itself. Only the person themselves or an admin can download the QR image.

QR Code Download

The fullscreen QR view (/profile/qr/) includes a Download button that saves the QR code as a PNG file named <name>_qr.png. This lets users save their QR code to their phone's camera roll for easy access at events.

Events

Creating Events

Organizers and admins can create events with the following fields: Name (required), Description, Location, Start/End Time (required), Registration Open/Close, Capacity (set to 0 for unlimited), and an optional External Link.

The event creator is automatically assigned as staff for their event. Email notifications for event creation and editing are sent only to the event owner.

Editing Events

Event owners can edit their events directly from the My Events page via an inline modal popup — no separate page navigation required. The modal pre-fills all event fields from the selected event and submits changes in place. Only the event owner (or an admin) can edit an event.

When an event is edited, registered attendees automatically receive an email notification with the updated details.

Organizer Event Management

Organizers and admins have a dedicated My Events page at /organizer_event_create/ that lists only events they have created. From this page they can:

  • View all their events — cards showing name, date, location, and registration count
  • Edit events inline — opens a blurred modal popup pre-filled with the event's current data
  • Navigate to event details — links to the full event detail page with attendee/staff/log tabs

This is separate from the admin dashboard — organizers see only their own events, while admins see everything.

Event Validation Rules

Events enforce the following date/time constraints on create and edit:

  • start_time must be before end_time
  • reg_open must be before reg_close
  • reg_close must be on or before start_time
  • reg_open must be before end_time

If reg_open or reg_close are left blank, they default to the event's creation time and start time, respectively. A capacity of 0 means unlimited.

Event Registration

Authenticated users can register for events through the event detail page or the API. Registration validates that the registration window is open, the event is not at capacity, and the user is not already registered. Canceled registrations can be reactivated.

Walk-In Mode

Event creators and organizers can toggle walk-in mode on an event. When enabled, scanning an unregistered person's QR code at check-in automatically registers them and checks them in — as long as the event isn't at capacity.

✓
Walk-in mode is ideal for informal events where pre-registration isn't required. Toggle it from the event detail page or via the API.

Event Status

Events expose computed status properties: Active (current time is between start and end), Upcoming (current time is before start), and Full (registration count has reached capacity).

Email Notifications for Events

The following emails are sent automatically during the event lifecycle:

  • Event creation — the event owner receives a confirmation email
  • Event editing — all registered attendees receive a notification with updated details
  • Registration confirmation — attendees receive an email when they register for an event
  • Reminders — registered attendees receive reminders at 24 hours and 1 hour before event start (configurable via REMINDER_WINDOWS in settings)

All email sending is wrapped in try/except — delivery failures never crash the application. Without SMTP configuration, emails print to the console for development.

CSV Export

Staff and above can download an attendee list for any event as a CSV file. The export is available via the Download CSV button on the dashboard event table or via the API at /api/events/<uuid>/attendees.csv.

The CSV includes columns: Name, Email, Organization, Phone, Status, Registered At, and Checked In At. Canceled registrations are excluded, and rows are sorted alphabetically by name.

Check-In & Scanning

The scanner page (/scanner/) is the core operational interface for event staff. It supports two modes.

Attendance Mode (Staff Only)

Used to check attendees into an event. Staff selects an active event, scans an attendee's QR code (or enters a UUID manually, or searches by name), the system validates registration, and a ScanLog is recorded regardless of outcome.

Discovery Mode (All Users)

Used for networking and contact exchange. Scanning a QR code in Discovery mode fetches the person's contact card, showing only the fields they've made visible through their privacy settings.

Scanner Features

  • Camera scanning — uses the device's rear-facing camera via the html5-qrcode library. Starts automatically when the scanner page loads (requires HTTPS for camera access)
  • Manual UUID entry — expandable fallback input for when the camera isn't available or a QR code is damaged. Paste or type the person's UUID and submit
  • Name search — staff can search attendees by name using the /api/people/search/ endpoint. Returns the top 10 matches with name and organization. Clicking a result triggers a check-in (Attendance mode) or loads their contact card (Discovery mode)
  • Scan cooldown — 2-second cooldown between scans prevents accidental double-scans from the camera reading the same QR code repeatedly
  • Scan history — recent scan results are displayed in a live feed on the page, showing the person's name, result status (success/duplicate/error), and timestamp
  • Result messages — color-coded feedback banners shown for 4 seconds after each scan: green for success, yellow for duplicates, red for errors, blue for informational messages
  • Event selection — in Attendance mode, staff must select an active event before scanning. The selected event persists in the server session across page loads
  • JWT authentication — the scanner page obtains a JWT token on load and uses it for all API calls, enabling direct fetch() requests to the REST endpoints

Profile Editing

Users manage their contact information from the profile page, which has two tabs:

  • Info tab — edit name, email, organization, phone, and links. Each field (except name) has a visibility toggle: "Visible when others scan your QR." The Links field is stored as JSON and supports multiple URLs (e.g., personal website, LinkedIn, GitHub)
  • QR & Tickets tab — shows the user's QR code with a link to the fullscreen view, and a list of their event registrations

Tickets

The profile page shows the user's event registrations (tickets) with event name, date, location, and ticket status. Ticket statuses are:

  • Issued — registered but not yet checked in
  • Checked In — successfully scanned at the event
  • Canceled — registration was canceled (can be reactivated)
  • Expired — event has ended

User Management (Admin Only)

Admins see a Users tab on the dashboard with full user management capabilities:

  • View all users — table showing username, name, email, and role for every account
  • Create users — add new accounts with username, password, full name, email, and role via a collapsible form

Event Detail Page

The event detail page (/events/<uuid>/) provides different views based on role:

  • All authenticated users — event info (name, description, location, times), stat cards (registered count, checked-in count, capacity), registration button, and an attendance ratio progress bar
  • Staff and above — additional tabbed interface:
    • Attendees tab — table of registered people with name, email, ticket status, registration time, and check-in time
    • Staff tab — list of assigned staff with a search-to-add autocomplete and remove buttons. Staff search queries users by name
    • Scan Logs tab — history of all scan attempts showing result code, person name, actor (who scanned), timestamp, and metadata
  • Event creator / admin — toggle for walk-in mode, staff management controls

Scan Logs

Every scan attempt is logged to the ScanLog model — successful or not — creating a complete audit trail. Each log entry records:

  • Event — which event the scan was for
  • Person — who was scanned (if valid)
  • Actor — which staff member performed the scan
  • Result — outcome code (success, duplicate, not_registered, invalid, wrong_event, event_inactive)
  • Scanned value — the raw UUID that was scanned
  • Metadata — JSON field with additional context (e.g., check-in timestamp for duplicates)
  • Timestamp — when the scan occurred

Admins can view all scan logs via the API at /api/logs with optional filtering by ?event=<uuid> or ?result=<code>. Logs are also visible on the event detail page's Scan Logs tab.

i
Scan logs are currently fetched on page load and on each new scan. Real-time streaming of logs (e.g. via WebSockets) is planned but not yet implemented.
Roles & Permissions

Turnstil uses four roles in ascending privilege.

RoleCapabilities
AttendeeRegister for events, manage profile, use Discovery mode scanner, view own QR code
StaffAll attendee capabilities + Attendance mode scanning (for assigned events), view dashboard, view attendee lists
OrganizerAll staff capabilities + create events, manage event staff, edit own events, toggle walk-ins
AdminFull system access — scan any event, view all logs, manage all events and staff, CRUD all user accounts, assign roles, register users for events
i
Staff must be explicitly assigned to an event (via the staff ManyToMany relationship) to scan check-ins for that event. Event creators are auto-assigned. Admins bypass this requirement.

API Permission Classes

The REST API enforces permissions using custom DRF permission classes:

ClassGrants Access To
IsStaffOrAboveUsers with role: staff, organizer, or admin
IsOrganizerOrAboveUsers with role: organizer or admin
IsAdminRoleUsers with role: admin only
IsEventStaffUsers who are the event creator, assigned staff, or admin for a specific event
PWA & Mobile Support

Turnstil supports installation as a Progressive Web App for a native app-like experience on mobile devices.

Features

  • Add to Home Screen — users can install Turnstil to their device's home screen via the browser's install prompt
  • Standalone mode — runs without browser chrome when launched from the home screen, providing a native app feel
  • App icons — 192×192 and 512×512 SVG icons with maskable purpose for Android adaptive icons
  • Apple support — includes apple-mobile-web-app-capable and apple-mobile-web-app-status-bar-style meta tags for iOS home screen integration
  • Theme color — status bar and app frame use #1c1c21 to match the dark UI

The PWA manifest is served at /static/manifest.json and referenced in the <head> of every page.

  • ?event=<uuid> — filter logs to a specific event
  • ?result=success|duplicate|not_registered|invalid|wrong_event|event_inactive — filter by result type

Each log entry includes: event, person_name, actor_name, result, scanned_value, metadata (JSON), and timestamp.

Data Model
Turnstil uses six Django models. Understanding the relationships between them helps when working with the API or extending the platform.
ModelPrimary KeyPurpose
UserAuto intExtends Django's AbstractUser with a role field (attendee, staff, organizer, admin). One-to-one link to Person via user.person
PersonUUIDIdentity and contact info. The UUID is the QR code payload. Fields: name, email, organization, phone, links (JSON), visibility (JSON)
EventUUIDEvent details with registration windows, capacity, walk-in toggle. Has a created_by FK to User and a staff M2M to User
TicketUUIDLinks a Person to an Event with status tracking. Unique constraint on (person, event) — one ticket per person per event
EventReminderAuto intTracks which reminder emails have been sent to prevent duplicates. Unique constraint on (event, hours_before)
ScanLogUUIDAudit trail of every scan attempt. Records event, person, actor (who scanned), result code, raw scanned value, and metadata (JSON)

Key Relationships

  • User → Person — one-to-one. Created together during registration. The Person holds the UUID/QR identity; the User holds login credentials and role
  • Person → Ticket → Event — many-to-many through Ticket. Each Ticket tracks registration status and check-in time
  • Event → User (staff) — many-to-many. Staff can scan check-ins for events they're assigned to. Event creators are auto-assigned
  • Event → ScanLog — one-to-many. Every scan attempt for an event is logged, whether it succeeds or fails

Ticket Status Lifecycle

  • ISSUED → CHECKED_IN — normal check-in flow via scanner
  • ISSUED → CANCELED — registration canceled
  • CANCELED → ISSUED — re-registration reactivates canceled tickets
  • ISSUED → EXPIRED — event has ended (not currently auto-set)
